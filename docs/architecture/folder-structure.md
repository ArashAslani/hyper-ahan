# Folder Structure Standard

**Status:** Permanent Phase 1 standard  
**Purpose:** Define ownership, imports, and boundaries for every major folder.

This project uses folders as architectural boundaries. A file belongs where its responsibility lives, not where it is shortest to import from.

---

## Root Folders

### `.ai/`

**Purpose:** Permanent AI operating instructions, workflows, and reusable prompts.

**Owner:** Lead frontend architect.

**Allowed content:**

- project constitution
- workflow checklists
- prompt templates

**Forbidden content:**

- implementation code
- generated feature output
- secrets

---

### `docs/`

**Purpose:** Human and AI documentation.

**Owner:** Lead frontend architect and future feature owners.

**Allowed content:**

- architecture standards
- development standards
- UI and SEO standards
- business and roadmap documentation
- audit reports

**Forbidden content:**

- duplicated source code
- outdated implementation instructions
- placeholder documents

Every major architectural decision must be reflected in `docs/`.

---

### `public/`

**Purpose:** Public static assets served directly by Next.js.

**Allowed content:**

- favicons
- manifest assets
- static images that must be public
- robots-adjacent static files only when not generated by Next.js

**Forbidden content:**

- source design files
- private uploads
- large unused asset dumps

Assets must have clear naming and business purpose.

---

### `src/`

**Purpose:** Application source code.

All production code lives under `src`.

---

## Source Folders

### `src/app/`

**Purpose:** Next.js App Router.

**Owns:**

- routes
- layouts
- route groups
- route metadata
- server/client composition
- framework files (`page.tsx`, `layout.tsx`, `loading.tsx`, `error.tsx`, `not-found.tsx`)

**Allowed imports:**

- `@/features/*`
- `@/services/*`
- `@/lib/*`
- `@/config/*`
- `@/types`
- `@/providers/*`

**Forbidden imports:**

- `@/mocks/*`
- another route's private implementation
- raw API calls outside services

**Example:**

```tsx
import { ProductDetailView } from "@/features/catalog/ProductDetailView";
import { productService } from "@/services/productService";
```

Pages should orchestrate; they should not become feature modules.

---

### `src/features/`

**Purpose:** Business-facing UI and feature logic.

**Owns:**

- feature views
- feature-specific components
- hooks
- constants
- feature utilities
- feature-local validation
- feature-local types

**Allowed imports:**

- own feature files
- `@/shared/ui/*`
- `@/services/*`
- `@/lib/*`
- `@/config/*`
- `@/types`

**Forbidden imports:**

- `@/mocks/*`
- another feature's private files unless explicitly exported as public API
- Next.js route internals from `src/app`

**Example allowed:**

```tsx
import { Button } from "@/shared/ui/Button";
import { routes } from "@/lib/routes";
```

**Example forbidden:**

```tsx
import { products } from "@/mocks/products";
```

---

### `src/shared/`

**Purpose:** Reusable code without business ownership.

Current primary child:

- `src/shared/ui/`

**Owns:**

- design-system primitives
- generic layout helpers
- reusable feedback components

**Allowed imports:**

- `@/lib/*` only when domain-neutral
- React
- framework-neutral types

**Forbidden imports:**

- `@/features/*`
- `@/services/*`
- `@/mocks/*`
- business-specific constants

Shared components must be reusable without understanding steel commerce.

---

### `src/services/`

**Purpose:** Data access boundary.

**Owns:**

- service functions
- request construction
- DTO mapping
- mock/API switching
- error normalization
- caching policy entry points

**Allowed imports:**

- `@/mocks/*`
- `@/lib/api-client`
- `@/types`
- service-local mappers/utilities

**Forbidden imports:**

- JSX
- `@/features/*`
- `@/shared/ui/*`
- browser-only UI logic

Services return domain-ready values for the UI.

---

### `src/mocks/`

**Purpose:** Temporary sample data for development until real API endpoints are connected.

**Allowed imports:**

- plain types if needed

**Forbidden imports:**

- `@/app/*`
- `@/features/*`
- `@/shared/*`
- `@/services/*` unless a mock factory explicitly needs a type-only import

Only services may consume mocks.

---

### `src/lib/`

**Purpose:** Small cross-cutting utilities.

**Owns:**

- routes
- storage helpers
- formatting helpers
- API client wrapper
- small pure utilities

**Allowed imports:**

- stable platform APIs
- primitive shared types

**Forbidden imports:**

- `@/features/*`
- `@/services/*` unless a utility is explicitly service-owned and better moved
- JSX

If a file in `lib` grows business-specific, move the rule to a feature, service, or domain module in a documented change.

---

### `src/config/`

**Purpose:** Static application configuration.

**Owns:**

- site metadata constants
- navigation configuration
- static display settings

**Allowed imports:**

- `@/lib/routes`
- simple types

**Forbidden imports:**

- services
- features
- mutable runtime state

Config is declarative. It must not perform I/O.

---

### `src/providers/`

**Purpose:** Global React context providers.

**Owns:**

- app-level context wrappers
- state that truly spans multiple routes

**Allowed imports:**

- `@/lib/*`
- `@/services/*` when global data is required
- `@/types`

**Forbidden imports:**

- page-specific UI
- feature internals that make all routes depend on one feature

Providers are expensive because they create client boundaries. Add a provider only when local composition cannot solve the problem.

---

### `src/types/`

**Purpose:** Shared domain type definitions.

**Allowed content:**

- stable domain entities
- reusable enums/unions
- DTO-independent view types

**Forbidden content:**

- component props that are local to one component
- backend DTOs unless clearly named and isolated
- runtime validation logic

Prefer feature-local types until a type is reused across boundaries.

---

## Import Rules

Use the `@/*` alias for source imports.

Allowed:

```ts
import { routes } from "@/lib/routes";
```

Avoid long relative paths across architecture boundaries:

```ts
import { routes } from "../../../lib/routes";
```

Relative imports are acceptable within a small local folder:

```ts
import { ProductRow } from "./ProductRow";
```

---

## Ownership Rule

Every file must answer:

1. What layer owns this file?
2. What business or technical decision does it represent?
3. Who is allowed to import it?
4. What would break if it changed?

If these answers are unclear, the file is in the wrong place or needs a better name.
